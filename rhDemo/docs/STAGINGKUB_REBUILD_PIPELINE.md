# Mise à jour en place de l'infrastructure stagingkub

## Objectif

Aujourd'hui, faire repartir `stagingkub` sur une base propre (pour absorber les mises à jour poussées par Renovate sur les briques d'infrastructure : image `kindest/node`, CNI Cilium, NGINX Gateway Fabric, stack Prometheus/Loki/Grafana) impose un enchaînement **manuel** sur la machine hôte :

1. `kind delete cluster --name rhdemo`
2. `rhDemo/infra/stagingkub/scripts/init-stagingkub.sh`
3. `rhDemo/infra/stagingkub/scripts/install-observability.sh`
4. Mise à jour manuelle du credential Jenkins `kubeconfig-stagingkub` (le token RBAC `jenkins-deployer` est régénéré à chaque `init-stagingkub.sh`)
5. Déclenchement de `Jenkinsfile-CD` pour redéployer l'application

L'idée est de porter cet enchaînement dans un pipeline Jenkins pour respecter le principe DevOps "l'infrastructure se reconstruit toute seule".

## Pourquoi ce n'est pas qu'un problème de scripting

Le point bloquant n'est pas technique mais **architectural / sécurité**, et il est documenté explicitement dans le dépôt :

- `Dockerfile.agent` (image des agents Jenkins éphémères) précise noir sur blanc : *"KIND NON INSTALLÉ - SÉCURITÉ RBAC (...) Pour gérer le cluster KinD, utilisez la machine hôte (pas Jenkins)"*.
- Les agents Jenkins ne montent jamais `/var/run/docker.sock` directement : ils passent par `docker-socket-proxy` (`rhDemo/infra/jenkins-docker/docker-compose.yml`), qui n'autorise que la création/suppression de **conteneurs** (agents éphémères) — `VOLUMES: 0`, `EXEC: 0`, `SWARM: 0`. Impossible de piloter `kind` (qui a besoin de créer des conteneurs *privilégiés*, des réseaux Docker dédiés, et de manipuler des volumes) à travers ce proxy filtré.
- Le kubeconfig `jenkins-deployer` (`credential kubeconfig-stagingkub`) est **scopé au namespace `rhdemo-stagingkub` + lecture de `monitoring`**, sans accès à `kube-system` ni aux namespaces système — donc incapable d'installer Cilium, NGINX Gateway Fabric, les CRD Prometheus Operator ou de créer les namespaces nécessaires. C'est le résultat volontaire de `infra/stagingkub/rbac/` (moindre privilège).
- `init-stagingkub.sh` a lui-même besoin d'un accès **cluster-admin** (il applique les RBAC, crée des namespaces, installe des CRD) + de `sudo` (écriture dans `/etc/hosts`) + du CLI `kind`.

Automatiser tel quel l'enchaînement manuel revient à redonner à Jenkins exactement les pouvoirs qu'on lui a délibérément retirés.

## Choix retenu et justification

**Mise à jour en place (`helm upgrade`) plutôt que reconstruction complète pilotée par Jenkins.** Deux alternatives ont été écartées avant de retenir cette approche : donner à Jenkins un agent privilégié (`docker.sock` + `kind` + kubeconfig cluster-admin) pour rejouer la séquence manuelle telle quelle, ou déléguer l'opération destructive à un script hôte planifié en parallèle de Jenkins. Les deux réintroduisent une surface d'attaque disproportionnée — un job compromis ou une PR Renovate malveillante pourrait détruire tout l'environnement — pour un besoin qui, dans les faits, ne concerne quasiment jamais une reconstruction totale mais des montées de version mineures/patch de composants déjà gérés par Helm.

Un vrai cluster Kubernetes de production n'est jamais détruit/recréé pour absorber une mise à jour de CNI, d'ingress ou d'observabilité : on fait un `helm upgrade`, exactement le modèle déjà utilisé pour l'application elle-même (`Jenkinsfile-CD` ne fait jamais de delete/recreate). Cette option généralise ce principe à l'infrastructure : chaque composant obtient un ServiceAccount dédié et strictement scopé par `resourceNames` (jamais cluster-admin, jamais `docker.sock`/`kind`), suit le même circuit Renovate + merge automatisé que les dépendances applicatives, et les scripts de reconstruction complète (`init-stagingkub.sh`, `install-observability.sh`, `clean-cluster.sh`) restent intacts et utilisables à la main pour le seul cas qui ne peut techniquement pas être traité en place : une montée de version de Kubernetes (`kindest/node`).

**Principe directeur pour la suite du document :** `init-stagingkub.sh`, `install-observability.sh` et `clean-cluster.sh` restent **strictement inchangés dans leur usage** — ils demeurent le chemin "reconstruction complète depuis rien", utilisable à la main exactement comme aujourd'hui (disaster recovery, montée de version majeure de Kubernetes, dérive de configuration). La mise à jour en place est un **chemin additionnel**, pas un remplacement. Pour éviter que les deux chemins divergent avec le temps, la logique Helm de chaque composant est extraite dans un script unique appelé par les deux chemins — voir étape 1.

## Étape 1 — Factoriser la logique d'installation par composant

Aujourd'hui, `init-stagingkub.sh` installe Cilium et NGINX Gateway Fabric avec un `if kubectl get ... ; then skip ; else helm install ...` (branche "déjà présent → ne rien faire"), donc rejouer le script sur un cluster existant ne met rien à jour. À l'inverse, `install-observability.sh` utilise déjà `helm upgrade --install` partout — il est donc, sans le savoir, déjà "prêt" pour la mise à jour en place.

Pour unifier les deux :

1. Créer `rhDemo/infra/stagingkub/scripts/components/` avec un script par brique, chacun idempotent (`helm upgrade --install`, jamais de test d'existence bloquant) :
   - `install-or-upgrade-cilium.sh` (extrait de `init-stagingkub.sh`, remplace le bloc `if ... else helm install ...` ; inclut la vérification de montée d'une seule version mineure à la fois et le job `preflight-check` recommandé par Cilium avant l'upgrade réel — voir étape 3)
   - `install-or-upgrade-ngf.sh` (extrait de `init-stagingkub.sh`, inclut la mise à jour des CRDs Gateway API via `kubectl kustomize ... | kubectl apply -f -`, qui est elle aussi idempotente)
   - `install-or-upgrade-kube-prometheus-stack.sh` (extrait d'`install-observability.sh` ; applique les CRDs avec `kubectl apply --server-side` avant le `helm upgrade`, voir étape 3 — un `kubectl apply` client-side échoue sur ces CRDs volumineuses)
   - `install-or-upgrade-loki.sh`, `install-or-upgrade-promtail.sh`, `install-or-upgrade-grafana.sh` (extraits quasi tels quels d'`install-observability.sh`, déjà en `helm upgrade --install`)
2. `init-stagingkub.sh` et `install-observability.sh` appellent ces scripts au lieu d'inliner les commandes Helm — leur comportement observable ne change pas (un cluster neuf obtient toujours les mêmes composants dans le même ordre).
3. Chaque script de composant passe d'un `helm upgrade --install ... --wait` à `helm upgrade --install ... --atomic --wait --timeout` : `--atomic` déclenche un rollback automatique si l'upgrade échoue, indispensable dès qu'un upgrade peut être lancé sans supervision humaine immédiate (aujourd'hui ce n'est pas la norme sur ces installs).

## Étape 2 — Rendre les versions traçables par Renovate

Aucune des versions d'infrastructure (Cilium, NGF, kube-prometheus-stack, Loki, Promtail, Grafana, digest `kindest/node`) n'est aujourd'hui suivie par Renovate — ce sont de simples variables shell/YAML. Il faut leur ajouter le même balisage `# renovate: datasource=... depName=...` que celui déjà utilisé dans `renovate.json` pour les images pré-épinglées de `Jenkinsfile-CI` :

- `install-or-upgrade-cilium.sh` : ajouter une variable `CILIUM_VERSION` (actuellement la valeur `1.18.6` est en dur dans la commande `helm install`, à extraire d'abord) + `customManager` `datasource=helm`, `depName=cilium`, `registryUrl=https://helm.cilium.io/`.
- `install-or-upgrade-ngf.sh` : `NGF_VERSION` déjà en variable → `customManager` `datasource=docker`, `depName=ghcr.io/nginx/charts/nginx-gateway-fabric` (pas de `registryUrl`). **Corrigé en production** : la première tentative utilisait `datasource=helm`/`registryUrl=oci://ghcr.io/nginx/charts`, mais le datasource `helm` de Renovate ne sait pas résoudre un `registryUrl` en `oci://` — il tente un `index.yaml` HTTP classique (format des dépôts Helm traditionnels), inexistant sur un registre OCI, et échoue avec `cannot parse url`/`Failed to look up helm package`. Pour un chart publié en OCI, le datasource `docker` avec le chemin complet du registre en `depName` fonctionne — même mécanisme que `NGF_IMAGE` dans `Jenkinsfile-CI`. Le digest `NGF_IMAGE_DIGEST` documenté en commentaire n'est utilisé nulle part dans la commande Helm (l'image est résolue par le chart) — à vérifier/nettoyer à cette occasion.
- Les CRDs Gateway API (`?ref=v${NGF_VERSION}` dans l'URL `kubectl kustomize`) : verrouillées sur la même variable `NGF_VERSION`, pas de suivi Renovate séparé nécessaire. En revanche, contrairement aux autres composants (tous installés via un chart Helm versionné et publié), ces CRDs sont aujourd'hui **récupérées en direct depuis GitHub à chaque exécution du script**, sur un tag (`?ref=vX`) sans épinglage par digest/commit — le seul point de tout `install-observability.sh`/`init-stagingkub.sh` qui ne suit pas la logique de reproductibilité (images/charts épinglés) déjà en place ailleurs dans le projet. **Implémenté : vendoring.** Le manifeste est commité dans le dépôt (`rhDemo/infra/stagingkub/gateway-api-crds/v<version>/crds.yaml`) au moment du bump de `NGF_VERSION`, via le script `scripts/vendor-gateway-api-crds.sh <NGF_VERSION>` (`kubectl kustomize ... > gateway-api-crds/v<version>/crds.yaml`) — exécuté et commité manuellement dans la même PR que le bump (pas encore automatisé dans un stage Jenkins ; à faire une fois le mécanisme éprouvé sur quelques cycles). Le manifeste v2.6.0 a été généré et vérifié à l'implémentation (8 CRDs Gateway API, cf. étape 3). Cela élimine la dépendance réseau live à GitHub pendant l'upgrade et conserve l'ancien manifeste pour un rollback propre (cf. étape 3).
- `install-or-upgrade-kube-prometheus-stack.sh` / `-loki.sh` / `-promtail.sh` / `-grafana.sh` : variables déjà présentes (`KUBE_PROMETHEUS_STACK_VERSION`, etc.) → `customManager` `datasource=helm` avec `registryUrl` correspondant (`https://prometheus-community.github.io/helm-charts`, `https://grafana.github.io/helm-charts`).
- `kind-config.yaml` (digest `kindest/node`) : réutiliser tel quel le `customManager` regex déjà défini dans `renovate.json`, en ajoutant simplement `rhDemo/infra/stagingkub/kind-config.yaml` à `managerFilePatterns`. Renovate ouvrira quand même la PR (visibilité via le Dependency Dashboard), mais elle sera traitée manuellement par reconstruction complète — voir étape 6, ce composant reste hors périmètre de l'upgrade en place.

## Étape 3 — RBAC dédié, étroitement scopé

**Décision actée : ServiceAccount dédié.** Toutes les extensions RBAC ci-dessous sont portées par un **nouveau** ServiceAccount `jenkins-infra-upgrader` (namespace `rhdemo-stagingkub`, comme `jenkins-deployer`), avec son propre credential Jenkins (`kubeconfig-stagingkub-infra-upgrader`), utilisé exclusivement par le job de mise à jour en place (étape 4). Le ServiceAccount `jenkins-deployer` et son credential `kubeconfig-stagingkub` existants, utilisés par `Jenkinsfile-CD`, restent **strictement inchangés** — ils ne gagnent aucun droit supplémentaire du fait de cette option. Cette séparation suit `rbac/setup-jenkins-rbac.sh`/`jenkins-serviceaccount.yaml` comme modèle à dupliquer (nouveau ServiceAccount + Secret de type token), pas à modifier.

`jenkins-infra-upgrader` couvre : `nginx-gateway`, `loki-stack`, `monitoring` (étendu au-delà des ServiceMonitors/PodMonitors actuels), `kube-system` (restreint par `resourceNames`, jamais un accès générique), plus les extensions `ClusterRole` détaillées ci-dessous. **Correction par rapport à une première intuition** : `jenkins-infra-upgrader` n'a aucun droit dans `rhdemo-stagingkub` — aucun des composants qu'il gère (Cilium, NGF, kube-prometheus-stack, Loki, Promtail, Grafana) n'y vit ; ce namespace reste exclusivement le domaine de `jenkins-deployer`/`Jenkinsfile-CD`. Plusieurs niveaux de risque, du plus simple au plus sensible :

- **Sans risque particulier — nouveaux `Role`/`RoleBinding` namespace par namespace pour `jenkins-infra-upgrader`**, calqués sur `jenkins-monitoring-role.yaml` :
  - `nginx-gateway` : Deployments, Services, ConfigMaps, Secrets (ressources créées par le chart NGF, toutes namespaced).
  - `loki-stack` : idem pour Grafana/Loki/Promtail (Deployments/StatefulSets/DaemonSets/Services/ConfigMaps/Secrets/PVC). Correction vérifiée à l'implémentation (`helm template`) : Loki et Grafana créent chacun, en plus, un petit `ClusterRole`/`ClusterRoleBinding` propre (`loki-clusterrole`, `grafana-clusterrole`) — strictement en lecture (`get/watch/list` sur `configmaps`/`secrets`, pattern de découverte par sidecar), à ajouter à la liste `resourceNames` du `ClusterRole` `jenkins-infra-upgrader-cluster-role` ci-dessous par cohérence, sans changer leur niveau de risque (toujours read-only).
- **Risque mesuré mais nécessaire — nouveau `ClusterRole` dédié** (`jenkins-infra-upgrader-cluster-role`, distinct de `jenkins-deployer-cluster-role`), pour les objets cluster-scoped que ces charts gèrent eux-mêmes :
  - `customresourcedefinitions` (`apiextensions.k8s.io`) : `get/list/watch` en lecture large (Helm doit pouvoir vérifier l'état des CRDs Gateway API, Prometheus Operator et Cilium), `update/patch` restreint par `resourceNames` aux seules CRDs des composants concernés — jamais de `create` libre, pour empêcher l'installation de CRDs arbitraires par ce biais.
  - `clusterroles`/`clusterrolebindings` (`rbac.authorization.k8s.io`) : `get/update/patch` restreint par `resourceNames` aux objets déjà créés par NGF/Prometheus Operator/Promtail/Cilium (ex. `ngf-nginx-gateway-fabric`, `prometheus-kube-prometheus-operator`, `promtail`, `cilium`, `cilium-operator`) — **jamais `create`**, pour empêcher la création de nouveaux ClusterRoleBindings arbitraires. Important : `jenkins-infra-upgrader-cluster-role` lui-même ne doit apparaître dans aucune de ces listes `resourceNames` — il ne doit jamais pouvoir se modifier lui-même.
  - Ces ajouts doivent être revus comme une exception documentée dans `infra/stagingkub/rbac/README.md`, pas comme une extension silencieuse du modèle de moindre privilège.

- **Cas particulier Cilium — géré, mais avec un garde-fou natif Kubernetes à exploiter explicitement.** Cilium est le CNI : DaemonSet `kube-system`, avec son propre `ClusterRole` très large (nécessaire pour que Cilium *lui-même*, via son ServiceAccount, lise pods/nodes/endpoints/services sur tout le cluster et programme le plan de données eBPF). La question n'est pas "Cilium a-t-il besoin de droits larges" (oui, structurellement, comme n'importe quel CNI) mais "`jenkins-infra-upgrader` doit-il pouvoir écrire dans ces objets pour faire l'upgrade" :
  - RBAC à ajouter, dans le même esprit que ci-dessus : Role scopé à `kube-system` avec `resourceNames` limités aux objets nommés de Cilium (`daemonset/cilium`, `deployment/cilium-operator`, `configmap/cilium-config`, `serviceaccount/cilium`, `serviceaccount/cilium-operator`, `secret/cilium-ca` si présent) — jamais un accès `kube-system` générique ; `clusterroles`/`clusterrolebindings` `cilium`/`cilium-operator` en `get/update/patch` uniquement, comme pour NGF/Prometheus.
  - **Correction vérifiée à l'implémentation, plus favorable que prévu** : aucun droit CRD `cilium.io` n'est nécessaire. Inspection du chart Cilium 1.18.6 (`helm template --include-crds`, `helm show crds`, `helm pull --untar`) : aucune CRD n'est rendue, aucun dossier `crds/` dans le chart. Cilium bootstrappe lui-même ses CRDs (`CiliumNetworkPolicy`, `CiliumEndpoint`, `CiliumIdentity`, `CiliumNode`, etc.) au démarrage de l'agent/operator via son propre ServiceAccount (dont le `ClusterRole` inclut déjà les droits nécessaires sur `customresourcedefinitions`) — `jenkins-infra-upgrader` n'a donc besoin de rien sur ce point, ce qui réduit d'autant le périmètre initialement envisagé.
  - **Le filet de sécurité réel** : Kubernetes empêche nativement l'auto-élévation de privilèges via RBAC — un sujet qui n'a que `update`/`patch` sur un objet `ClusterRole` (sans le verbe spécial `escalate`) ne peut pas réécrire les règles de cet objet pour lui accorder plus de droits qu'il n'en détient déjà lui-même. Concrètement : `jenkins-infra-upgrader` n'a et n'aura jamais de droits proches de ceux du ClusterRole `cilium` (lecture cluster-wide de pods/nodes/endpoints/NetworkPolicies) — si une nouvelle version de Cilium a besoin d'étendre les règles de son propre ClusterRole (ça arrive surtout sur des montées de version majeures/mineures ajoutant une fonctionnalité), l'API server refusera la modification, l'étape Helm échouera, et `--atomic` déclenchera un rollback automatique. Dans ce cas précis (rare), l'upgrade Cilium repasse en traitement manuel/admin — mais le mode d'échec est un blocage propre et tracé, pas une élévation de privilège silencieuse.
  - **Décision actée : pas de garde-fou humain supplémentaire.** Cilium suit le même automerge patch/minor que les autres composants (pas d'équivalent `dependencyDashboardApproval` spécifique) — le filet de sécurité `escalate` + `--atomic`/preflight-check ci-dessus est jugé suffisant.
  - Contrainte propre à Cilium à respecter dans `install-or-upgrade-cilium.sh` : suivre la procédure de montée de version documentée par le projet (upgrade une version mineure à la fois, jamais de saut ; exécuter le job de vérification `cilium preflight-check`/`cilium-cli` avant l'upgrade réel) plutôt qu'un simple `helm upgrade` brut — le DaemonSet redémarre alors composant par composant sans coupure réseau prolongée (un seul nœud control-plane sur KinD, donc une brève interruption du plan de données est à attendre pendant le rolling restart, à faire hors heures d'usage si le calendrier régulier le permet).
  - **Correction vérifiée à l'implémentation, point manqué initialement : le stockage de release Helm lui-même.** `helm list`/`helm upgrade --install` ont besoin de `list`/`get` sur leurs propres secrets de suivi (`sh.helm.release.v1.cilium.v<N>`), dont le nom change à chaque révision — impossible à couvrir par `resourceNames`. Erreur constatée en pipeline réel : `secrets is forbidden ... in the namespace "kube-system"`. Plutôt qu'élargir `jenkins-infra-upgrader-kube-system-role.yaml` à un accès secrets générique (exposerait tokens bootstrap, certs kubeadm...), la release Cilium utilise `--set namespaceOverride=kube-system` avec `--namespace cilium-release` : les ressources réelles (DaemonSet, ConfigMap, ServiceAccounts) restent dans `kube-system` sous les droits nommés existants, seul le stockage Helm vit dans ce namespace dédié à usage unique (`jenkins-infra-upgrader-cilium-release-role.yaml`, secrets sans `resourceNames` — sans risque car namespace mono-usage, même logique que `nginx-gateway`/`loki-stack`).

- **Cas particulier CRDs Gateway API / NGF — même logique, mais sans le filet de sécurité natif dont bénéficie Cilium.** Il faut distinguer deux choses aujourd'hui mélangées dans la réflexion : le RBAC sur les **instances** (`Gateway`, `HTTPRoute`, `ReferenceGrant`, `ClientSettingsPolicy`, `ProxySettingsPolicy`, `ObservabilityPolicy`, `RateLimitPolicy`) est déjà couvert par `jenkins-role.yaml` (ressources namespaced dans `rhdemo-stagingkub`) ; ce qui manque, c'est le droit de mettre à jour la **définition** de ces CRDs elles-mêmes (objets `CustomResourceDefinition`, cluster-scoped), nécessaire quand une nouvelle version de NGF/Gateway API ajoute un champ ou une ressource.
  - RBAC à ajouter sur `customresourcedefinitions` (`apiextensions.k8s.io`), `get/list/watch` en lecture, `update/patch` restreint par `resourceNames` à la liste exacte et nommée des CRDs déjà installées. Liste vérifiée à l'implémentation (`kubectl kustomize` sur le manifeste vendoré + `helm show crds` sur le chart NGF réel, plus complète qu'une première estimation) : channel standard Gateway API (8) — `backendtlspolicies`, `gatewayclasses`, `gateways`, `grpcroutes`, `httproutes`, `listenersets`, `referencegrants`, `tlsroutes` (`.gateway.networking.k8s.io`) ; CRDs propres à NGF embarquées dans le chart (11) — `authenticationfilters`, `clientsettingspolicies`, `nginxgateways`, `nginxproxies`, `observabilitypolicies`, `proxysettingspolicies`, `ratelimitpolicies`, `snippetsfilters`, `snippetspolicies`, `upstreamsettingspolicies`, `wafpolicies` (`.gateway.nginx.org`).
  - **Différence importante avec Cilium : pas de garde-fou `escalate` ici.** La protection anti-élévation de Kubernetes ne s'applique qu'aux objets `Role`/`ClusterRole`/`RoleBinding`/`ClusterRoleBinding` — pas aux `CustomResourceDefinition`. La seule barrière réelle est donc la liste `resourceNames` elle-même, associée au refus explicite de `create` : un `jenkins-deployer` qui aurait `create` libre sur `customresourcedefinitions` pourrait définir n'importe quel type de ressource cluster-wide, ce qui n'est pas souhaitable même si cela ne confère pas automatiquement de droits sur les instances de ce nouveau type. `create` reste donc **hors du périmètre accordé**.
  - Conséquence directe : si une nouvelle version de NGF introduit un **nouveau type de CRD** (un nouveau nom, absent de la liste `resourceNames` ci-dessus — c'est arrivé par le passé, ex. apparition de `ObservabilityPolicy`/`RateLimitPolicy` dans des versions récentes de NGF), l'étape d'application de la CRD échoue proprement (`Forbidden`, pas de `create`), l'upgrade s'arrête avant le `helm upgrade` (voir ordre ci-dessous), et le traitement redevient manuel pour cette version précise : ajout du nouveau nom à `resourceNames` (revue humaine du RBAC, cf. `infra/stagingkub/rbac/README.md`), puis rejeu du pipeline. Les montées de version qui n'introduisent pas de nouveau type de CRD (l'écrasante majorité des patches/minors) passent, elles, en automatique.
  - Ordre d'application à respecter dans `install-or-upgrade-ngf.sh` (déjà l'ordre actuel du script, à conserver) : appliquer les CRDs **avant** le `helm upgrade --install ngf ...`, jamais après — NGF s'attend à trouver le nouveau schéma dès son démarrage/rollout.
  - Piège Helm à couvrir explicitement : les CRDs `gateway.nginx.org` sont embarquées dans le répertoire `crds/` du chart NGF (confirmé via `helm show crds`), et Helm ne les met **jamais** à jour lors d'un `helm upgrade` — c'est une limitation connue et documentée de Helm (le contenu de `crds/` n'est appliqué qu'au premier `helm install`). Elles sont donc extraites et appliquées explicitement par `kubectl apply`, exactement comme pour les CRDs Gateway API standard, plutôt que de compter sur le `helm upgrade` pour le faire.
  - Validation avant merge (même mécanique que le `--dry-run` déjà prévu à l'étape 4 pour les releases Helm) : `kubectl apply --dry-run=server -f <manifeste-crds>` pour repérer une incompatibilité de schéma avant de merger la PR, sans jamais muter le cluster à ce stade.
  - Rollback : contrairement à une release Helm (`--atomic` suffit), un `kubectl apply` de CRD n'est pas capturé par `helm rollback`. Revenir en arrière signifie ré-appliquer l'ancien manifeste — possible proprement grâce au vendoring des manifestes par version (cf. étape 2), pas s'ils n'étaient que récupérés en direct depuis GitHub à la volée.

- **Cas particulier kube-prometheus-stack — le composant le plus composite du lot, à traiter sous-partie par sous-partie.** Une seule release Helm regroupe en réalité quatre éléments de risque distincts (visibles dans `prometheus-values.yaml`) : l'Operator lui-même, les objets `Prometheus`/`Alertmanager` qu'il réconcilie, Kube State Metrics, et Node Exporter. `kubeControllerManager`/`kubeScheduler`/`kubeProxy`/`kubeEtcd` sont déjà désactivés dans les values ("pas accessible sur KinD") et le sous-chart Grafana est désactivé (`grafana.enabled: false`, le Grafana de `loki-stack` est réutilisé) — ces deux points réduisent déjà la surface par rapport à une installation par défaut du chart.
  - **Prometheus Operator** : Deployment namespaced, mais avec son propre `ClusterRole` large (nécessaire pour qu'il puisse lire/réconcilier des ressources dans tous les namespaces et gérer les CRDs `monitoring.coreos.com`). Même traitement que Cilium/NGF : extension du `ClusterRole` `jenkins-infra-upgrader-cluster-role` sur `customresourcedefinitions` (les 10 CRDs `monitoring.coreos.com` : `alertmanagerconfigs`, `alertmanagers`, `podmonitors`, `probes`, `prometheusagents`, `prometheuses`, `prometheusrules`, `scrapeconfigs`, `servicemonitors`, `thanosrulers` — liste vérifiée par `helm show crds`) et sur `clusterroles`/`clusterrolebindings` nommés. Vérification `helm template` : le chart en crée en réalité **trois**, pas deux — `prometheus-kube-prometheus-operator`, `prometheus-kube-state-metrics`, et `prometheus-kube-prometheus-prometheus` (ClusterRole propre du serveur Prometheus lui-même). `get/update/patch` sur ces trois objets, jamais `create` — protégé par le même garde-fou `escalate` que Cilium pour les objets RBAC, mais **pas** pour les CRDs (même limitation que Gateway API : la seule barrière est `resourceNames` + refus de `create`).
  - **Piège Helm spécifique à ce chart, documenté dans `install-or-upgrade-kube-prometheus-stack.sh`** : les CRDs de kube-prometheus-stack sont volumineuses au point de dépasser la limite de taille de l'annotation `kubectl.kubernetes.io/last-applied-configuration` utilisée par un `kubectl apply` classique (client-side) — un problème connu et documenté du projet. Elles sont appliquées avec `kubectl apply --server-side` (`--force-conflicts` en cas de changement de propriétaire de champ entre deux versions), jamais un `kubectl apply` nu ni un simple `helm upgrade` en espérant qu'il gère les CRDs.
  - **Objets `Prometheus`/`Alertmanager`/`PrometheusRule` (CR instances, namespaced dans `monitoring`)** : contrairement aux CRDs elles-mêmes, ce sont des ressources namespaced sans risque cluster-scoped — mais `jenkins-monitoring-role.yaml` (celui de `jenkins-deployer`) ne couvre que `servicemonitors`/`podmonitors`. Un Role distinct pour `jenkins-infra-upgrader` couvre en plus `prometheuses`, `alertmanagers`, `prometheusrules`, `probes`, `thanosrulers`, `alertmanagerconfigs`, `scrapeconfigs`, en CRUD namespaced classique.
  - **Admission webhooks (`prometheusOperator.admissionWebhooks`)** : **implémenté** — `prometheusOperator.admissionWebhooks.enabled: false` ajouté dans `prometheus-values.yaml`. Vérifié par `helm template` avant/après : le chart créait un `ClusterRole`/`ClusterRoleBinding` `prometheus-kube-prometheus-admission` et des objets `ValidatingWebhookConfiguration`/`MutatingWebhookConfiguration` (`admissionregistration.k8s.io`, cluster-scoped) avec la valeur par défaut du chart ; les trois disparaissent une fois l'option désactivée. Aucun RBAC `admissionregistration.k8s.io` n'est donc nécessaire pour `jenkins-infra-upgrader`.
  - **Kube State Metrics** : `ClusterRole` large mais **strictement en lecture** (`get/list/watch`, aucun verbe d'écriture) sur la quasi-totalité des objets de l'API — risque bien moindre que les ClusterRoles d'écriture ci-dessus ; même traitement RBAC (resourceNames sur son propre ClusterRole/ClusterRoleBinding) par cohérence, mais le pire cas d'un abus reste une fuite de visibilité en lecture seule, pas un contrôle du cluster.
  - **Node Exporter** : DaemonSet, mais à l'inverse de tous les composants précédents, **son risque n'est pas RBAC** — il n'a quasiment aucun droit d'API Kubernetes (il ne fait que lire `/proc`/`/sys` sur l'hôte via `hostNetwork`/`hostPID`/hostPath). L'upgrade en place ne change rien à ce risque : `jenkins-infra-upgrader` obtient seulement le droit de mettre à jour la version d'image du DaemonSet (droits `apps/daemonsets` génériques du Role), jamais un accès au contenu qu'il lit sur les nœuds.

- **Cas particulier Promtail — le plus simple et le moins risqué de tous les composants traités ici.** Chart déployé dans `loki-stack`, un seul DaemonSet, aucune CRD propre :
  - RBAC : `ClusterRole` standard du chart, **strictement en lecture** (`get/list/watch` sur `pods`/`namespaces`/`nodes`, aucun verbe d'écriture) pour la découverte de service Kubernetes (`kubernetes_sd_configs`) — même traitement RBAC par cohérence (`clusterroles`/`clusterrolebindings` nommé `promtail`, `get/update/patch`, jamais `create`), mais l'impact d'un abus resterait une lecture de métadonnées, jamais une action d'écriture.
  - Le risque réel de Promtail n'est de toute façon pas lié à cette étude : `promtail-values.yaml` restreint déjà la découverte de service aux pods du namespace `rhdemo-stagingkub` (`kubernetes_sd_configs.namespaces.names: [rhdemo-stagingkub]`), mais le pod Promtail monte quand même `/var/log/pods` de l'hôte, donc techniquement lisible pour tous les namespaces au niveau disque — ce point est indépendant du pipeline d'upgrade (`jenkins-infra-upgrader` n'a et n'obtient aucun droit `pods/exec` dans `loki-stack`, donc aucune capacité nouvelle d'aller lire ce contenu depuis Jenkins).
  - Aucune CRD, aucun webhook d'admission, aucun `ClusterRole` en écriture : c'est le composant le plus sûr à confier au pipeline de mise à jour en place, avec un simple `helm upgrade --atomic --wait`, sans étape de préflight ni contrainte de saut de version.

Avec ces ajouts, **seule la version de Kubernetes elle-même (`kindest/node`) reste hors périmètre de la mise à jour en place** (étape 6) — Cilium, les CRDs Gateway API/NGF, kube-prometheus-stack et Promtail suivent le même chemin que Loki/Grafana, chacun avec son garde-fou documenté et son niveau de risque explicité (RBAC nommé partout ; protection `escalate` native pour les objets `ClusterRole`/`ClusterRoleBinding` ; liste `resourceNames` + refus de `create` pour les CRDs et les webhooks d'admission, sans équivalent natif à `escalate`) plutôt qu'une exclusion de principe.

## Étape 4 — Déclenchement du job `RHDemo-Stagingkub-Upgrade-Deploy` après merge

> **Aiguillage de la validation Renovate** (quelle PR reçoit quelle validation — Maven+OWASP,
> Kubernetes dry-run, ou Trivy scan ciblé — et selon quels fichiers modifiés) : voir
> [RENOVATE_AUTOMERGE_CI.md](RENOVATE_AUTOMERGE_CI.md), section « Aiguillage de la validation selon
> les fichiers modifiés ». Ce document-ci ne couvre que ce qui concerne spécifiquement le pipeline
> `RHDemo-Stagingkub-Upgrade-Deploy` lui-même (déclenché post-merge pour les PR de composants
> d'infra stagingkub).

**Validation avant merge, application réelle après merge** — même séparation que le couple
`Jenkinsfile-CI`/`Jenkinsfile-CD` pour l'application (une PR non encore mergée ne déploie jamais sur
stagingkub) :

1. *Après le merge réussi* d'une PR de composant d'infra : déclenchement du job
   `RHDemo-Stagingkub-Upgrade-Deploy` (`Jenkinsfile-Stagingkub-Upgrade-Deploy`, déclaré en JCasC dans
   `jenkins-casc.yaml`, sur le même modèle que `RHDemo-CI`/`RHDemo-CD`/`RHDemo-Renovate`), via
   `build job: 'RHDemo-Stagingkub-Upgrade-Deploy', wait: false` juste après l'appel de merge Forgejo
   réussi. Ce job exécute le vrai `install-or-upgrade-<composant>.sh` sur stagingkub (`--atomic` déjà
   intégré dans le script, cf. étape 1 — pas un argument à passer), avec rollback Helm automatique en
   cas d'échec, puis les mêmes vérifications que `Jenkinsfile-CD` (`kubectl rollout status`, health
   check applicatif — en particulier revérifier que `rhdemo-app` et `keycloak` restent joignables
   après un upgrade NGF, le Gateway partagé étant un point de passage commun à toute l'application).
2. Ce job utilise le credential dédié `kubeconfig-stagingkub-infra-upgrader` (ServiceAccount
   `jenkins-infra-upgrader` — jamais `kubeconfig-stagingkub`/`jenkins-deployer`, réservé à
   `Jenkinsfile-CD`, cf. étape 3).
3. Ce job tourne sur le **même agent éphémère que les autres pipelines** (pas de `docker.sock`,
   toujours pas de `kind` installé) — cette option n'élargit jamais le modèle d'agent, uniquement la
   portée RBAC Kubernetes, de façon nommée et auditée.

## Étape 5 — Validation avant activation de l'automerge

Avant de brancher l'automerge Renovate sur ces fichiers, dérouler manuellement au moins un cycle complet (ex. bump mineur de Grafana) et comparer l'état obtenu à celui d'une reconstruction complète (`clean-cluster.sh` + `init-stagingkub.sh` + `install-observability.sh` sur un cluster de test) : mêmes versions de charts, mêmes CRDs, aucune régression sur les dashboards/alertes. Ce n'est qu'après cette vérification que l'automerge patch/minor peut être activé sur ces composants, comme pour les dépendances applicatives.

## Étape 6 — Ce qui reste exclusivement du ressort de la reconstruction complète

- **`kindest/node`** (version de Kubernetes) : `kind` ne supporte pas le remplacement d'image de nœud en place ; changer de version mineure/majeure de Kubernetes impose `kind delete` + recréation. C'est le **seul** cas exclu de la mise à jour en place.
- **Couplage à surveiller** : une montée de version de Kubernetes s'accompagne parfois d'une contrainte de version minimale côté Cilium (compatibilité kube-proxy replacement/eBPF avec la nouvelle API serveur) — dans ce cas précis, la mise à jour de Cilium qui accompagne la reconstruction complète se fait via `init-stagingkub.sh` (pas via le pipeline d'upgrade en place), pour rester cohérente avec la nouvelle version de nœud installée au même moment.
- **Toute dérive d'état** (résidus de test, secrets corrompus, cluster dans un état incohérent) : reste couverte par `clean-cluster.sh` + `init-stagingkub.sh` + `install-observability.sh`, strictement inchangés, utilisables à la main pour ces cas rares — la mise à jour en place ne remplace jamais la capacité de tout reconstruire.
